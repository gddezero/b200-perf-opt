# B200 大模型推理性能评测报告

> **测试时间**：2026 年 3 月 | **硬件**：GCP A4 B200 SXM × 8 (180 GB HBM/卡)
> **软件**：vLLM 0.17.1rc1 nightly (cu130) | **压测工具**：evalscope perf
> **标准负载**：4500 token 输入，~175 token 输出，ignore_eos=true，无 prefix cache

---

## 1. 核心结论

### 1.1 B200 vs H200 性能对比（FP8，无 MTP，同条件公平对比）

| 模型 | B200 测试编号 | TTFT 倍率 (c=1) | 吞吐倍率 @40 (RPS) | toks/s 倍率 @40 |
|------|:---:|:---:|:---:|:---:|
| Qwen3-235B | #98 | **2.5×** (139ms vs 349ms) | **1.7×** (5.97 vs 3.57) | **1.7×** (1194 vs 715) |
| Qwen3.5-397B | #136 | **—** | **2.1×** (6.79 vs 3.23) | **2.1×** (1358 vs 647) |
| DeepSeek V3.1 | #120 | **—** | **2.5×** (3.53 vs 1.39) | **2.5×** (706 vs 277) |
| DeepSeek V3.2 | #105 | **—** | **2.3×** (3.21 vs 1.42) | **2.3×** (643 vs 284) |

> H200 数据来源于客户提供的同等 8 卡部署实测。

**关键结论**：
1. **吞吐量全面翻倍**：中高并发 (≥20) 下，B200 超越 H200 **1.7~2.5 倍**
2. **延迟大幅压缩**：V3.2 低延迟配置首 token 仅 **335ms**
3. **量化不影响吞吐**：FP8 与 BF16 吞吐持平，NVFP4 在 FP8 基础上再提升 **31~65%**

### 1.2 叠加 NVFP4 量化（SM100 独有能力）

| 模型 | FP8 @40 toks/s (编号) | NVFP4 @40 toks/s (编号) | 提升 | vs H200 综合倍率 |
|------|---:|---:|:---:|:---:|
| Qwen3-235B | 1194 (#98) | 1565 (#87) | **+31%** | **2.2×** |
| Qwen3.5-397B | 1358 (#136) | — | — | — |
| DeepSeek V3.1 | 706 (#120) | 1164 (#125) | **+65%** | **4.2×** |
| DeepSeek V3.2 | 643 (#105) | 967 (#115) | **+50%** | **3.4×** |

### 1.3 叠加 MTP 投机解码（低并发额外加速，n=2 推荐）

| 模型 | n=0 c=1 toks/s | n=2 c=1 toks/s | 提升 | TPOT 降低 |
|------|---:|---:|:---:|:---:|
| Qwen3.5-397B FP8 | 137 | 207 | **+52%** | 6→4ms |
| DeepSeek V3.1 FP8 | 115 | 178 | **+55%** | 8→5ms |
| DeepSeek V3.2 FP8 | 69 | 117 | **+69%** | 13→7ms |

> Qwen3-235B 模型架构不支持 MTP。n=2 为所有模型低并发场景的推荐值。

### 1.4 LMCache KV Cache Offload（长上下文多轮对话）

> 场景：V3.2 FP8，avg 84k token 输入，30 轮对话，c=48，LMCache L1=**1600 GB** (DRAM)

| 指标 | LMCache OFF | LMCache ON | 提升 |
|------|---:|---:|:---:|
| RPS | 0.223 | 2.200 | **+9.87×** |
| TTFT avg | 13,906ms | 1,177ms | **11.8× 降低** |
| TTFT p99 | 58,012ms | 4,037ms | **14.4× 降低** |

96% 的 prompt token 由 KV 缓存提供，仅 4% 需要实际计算。

## 3. TP/DP/EP 并行策略规律

### 3.1 通用规律

| 配置 | 吞吐 | 延迟 | 适用场景 |
|------|------|------|---------|
| TP=1 DP=8 EP=on | **最高** | 较高 | 高并发高吞吐 |
| TP=2 DP=4 EP=on | 高 | 中等 | 均衡 |
| TP=4 DP=2 EP=on | 中等 | 较低 | — |
| TP=8 DP=1 EP=off | 最低 | **最低** | 低延迟、单请求 |

**核心权衡**：TP 增大 → TTFT 降低（更多卡分担 prefill）→ 但吞吐下降（batch 能力减弱）

### 3.2 DeepSeek V3.2 注意力后端对比

| TP | FlashMLA vs FlashInfer |
|----|----------------------|
| TP=1 | FlashInfer 优 (+7%) |
| TP=2 | FlashMLA 略优 (+5%) |
| TP=4 | FlashInfer 显著优 (+15%) |
| TP=8 | FlashInfer 优 (+12%) |

> FlashInfer 在多数 TP 配置下更优，推荐作为默认选择。

### 3.3 MTP 投机解码规律

- **低并发 (c≤4) 收益显著**：n=2 vs n=0 提升 52~69%
- **高并发 (c≥40) 收益趋零**：batch 处理已摊薄自回归开销
- **推荐 n=2**：一致优于 n=1，n=3 因模型而异（可能负向）
- **不支持 MTP 的模型**：Qwen3-235B

## 4. 压测方法论

### 4.1 标准压测参数

| 参数 | 值 |
|------|-----|
| 工具 | evalscope perf (OpenAI Chat API) |
| 输入长度 | 4500 token (固定) |
| 输出长度 | ~175 token (ignore_eos=true) |
| 并发梯度 | 1 → 2 → 4 → 8 → 20 → 40 → 60 |
| Prefix Cache | 关闭 |
| 成功率要求 | 100% |

### 4.2 指标说明

| 指标 | 含义 |
|------|------|
| RPS | 每秒完成请求数 |
| TTFT | 首 token 延迟 (Time To First Token) |
| TPOT | 生成阶段每 token 延迟 (Time Per Output Token) |
| toks/s | 每秒生成 token 数 (= RPS × avg output tokens) |
| Avg Lat | 端到端平均延迟 |

### 4.3 多轮对话压测

| 参数 | 值 |
|------|-----|
| 工具 | benchmark_serving_multi_turn.py |
| 客户端数 | 40~50 |
| 最大活跃对话 | 150~500 |
| 最大轮次 | 30 |
| 共享前缀 | 8k tokens (system prompt) |
| 对话历史 | lognormal avg=30k, max=120k tokens |
